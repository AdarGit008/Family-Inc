### 5. GAPFILL_MAX_BATCH overflow is permanently uncategorized, not retried next run as the comment claims
- **Severity:** major  ·  **Dimension:** finance-ingest  ·  **Verdicts:** confirmed(minor), confirmed(major)
- **Where:** `automation/lib/categorize.py:34-37`
- **Contract:** SPEC §12.2 ("DeepSeek assigns categories the rules miss") + correctness; the comment documents behavior the code does not have
- **Claim:** The cap comment asserts self-healing: "the overflow stays blank this run and gets a fresh try next run (the dedup means only genuinely-new txns are ever re-presented)." This is false and inverted. After ingest, the overflow (rules-miss txns 81..N) are WRITTEN to Finance-Transactions with a blank Category but a real Txn-ID. On the next run, finance_ingest.run() builds `seen` from EVERY Txn-ID already on the tab (finance_ingest.py:256-257), so those Txn-IDs match, the rows go to `txns_seen` and are excluded from `new_txns` (finance_ingest.py:261-265). They are never re-presented to categorize_transactions, so the LLM never gets a second chance — they stay blank permanently until a human edits them. This is most likely to bite on the very first live import, where a 45-day backlog (WINDOW_DAYS=45 in scrape.js:40) of rules-miss transactions can easily exceed 80, silently leaving the tail uncategorized forever and absent from every Finance-Budget SUMIFS bucket.
- **Evidence:** categorize.py: "# A rules-miss batch sent to the LLM is capped ... the overflow stays blank this run and gets a fresh try next run (the dedup means only genuinely-new txns are ever re-presented). GAPFILL_MAX_BATCH = 80"; finance_ingest.py:256 `seen = {str(v).strip() for v in sheet.read_column(cfg.FINANCE_TRANSACTIONS_TAB, "Txn-ID", sheet_path)}`; finance_ingest.py:261 `if tid in seen or tid in batch: res.txns_seen += 1; continue`; _gapfill: `batch = misses[:GAPFILL_MAX_BATCH]`
- **Fix:** Either (a) loop the gap-fill in chunks of GAPFILL_MAX_BATCH within a single run so all rules-misses are categorized before the write, or (b) fix the comment to state the overflow is permanently blank until human edit, and add a re-categorize path that re-presents already-written blank-Category rows. Option (a) is the minimal correct fix; raise the cap or chunk-loop in _gapfill.

### [high] Cross-lane category vocabulary contract: categorizer rules CSV (seeds/14_Finance_Category_Rules.csv) vs Finance-Budget tab category labels (the SUMIFS keys)
The two vocabularies are independently maintained and DIVERGE today, with no test, no runtime guard, and no canon contract checking categorizer.vocabulary() subset budget.categories. Rules emit {Dining, Fees, Shopping, Income, ...}; the committed Family_OS.xlsx Finance-Budget rows are {Dining out, Subscriptions, Savings, Other, Health, ...}. This is silent money-loss in the weekly briefing's budget reconciliation AND the dashboard money screen — and it lands exactly on the M6.4 reconciliation the project is about to validate (gated ~2026-06-20). The seed CSV even self-documents the hazard ('PROVISIONAL: the category set mirrors the seed Finance-Budget rows; Shanee's budget migration is the vocabulary authority and may rename/extend these'). No dimension cross-checked the two vocabularies.

### [high] Engine/digest -> Sheet stamp ordering: daily_digest stamps Last Sent/Status (and clears the fail flag) immediately after QUEUEING to the outbox JSONL, not after the bridge confirms send
daily_digest.run (lines ~349-367) calls stamp_sent + _clear_fail_flag once outbox.queue() returns res.queued — but queue() only appends a row to whatsapp_outbox.jsonl; actual delivery happens later in the bridge's 15s poll, which can REFUSE the row (refused_unknown_recipient) or never deliver (permanent disconnect). The outbox-budget dimension covered the budget ledger but not this queue-vs-deliver acknowledgement seam; there is no path that reconciles whatsapp_sent.jsonl status back to the Sheet stamp.

### [medium] Unbounded append-only outbox/sent JSONL growth + O(n) per-poll rescan (automation/lib/outbox.py + bridge processOutbox)
whatsapp_outbox.jsonl and whatsapp_sent.jsonl are append-only and NEVER compacted/rotated (grep found no truncate/rotate/unlink for either). Every bridge poll (every 15s, 24/7) does readFileSync of the full outbox and rebuilds the entire SENT dedup Set; every Python queue() rebuilds _seen_pairs over the whole history to enforce idempotency. Over months on a small VPS this is monotonically growing memory/IO per poll and per queue, and the idempotency index is the budget/dedup correctness backbone. No dimension probed long-run state-file growth or the cost of the cross-process idempotency index.

### [medium] Duplicated recurrence logic across surfaces: automation/lib/dates.bump_due (Python engine) vs dashboard/app.js bumpDate() (JS) — 'must not diverge' contract with no enforcement
dates.py and app.js both carry the explicit comment that the two implementations mirror each other and must stay in sync (Weekly/Monthly/Quarterly/Yearly + Feb-29 month-end clamp). They are hand-duplicated in two languages with no shared source and no cross-check test. If the dashboard's bumpDate clamps differently from add_months (e.g. leap-year edge, or the dashboard writing the bumped Due Date in a different date format — already a disputed finding at app.js:1234), the engine and dashboard will disagree on the next due date for the same Done recurring row, and one will silently overwrite the other. No dimension diffed the two recurrence implementations behaviorally.

### [medium] Seed-backend formula-read semantics: openpyxl data_only=True returns None for never-recomputed formula cells (Reminders K/L Days-Until/Auto-flag; Finance-Budget Actual SUMIFS)
lib/sheet's seed backend reads with data_only=True, which yields the LAST CACHED value of a formula or None if openpyxl wrote the file (openpyxl never evaluates formulas). The committed Family_OS.xlsx shows Finance-Budget 'actual' = None for every row. Any test or creds-less dev run that relies on K/L or budget actuals silently sees None/0 rather than the real computed value, so tests can pass against zeros that would never occur live (this is exactly why test_finance re-implements SUMIFS — a disputed finding, but the root cause is unexamined). The tests-quality dimension flagged the re-implementation symptom; no dimension flagged that the seed backend cannot represent the formula columns at all, making a whole class of formula-dependent behavior untestable offline.

### [medium] Hebcal client (automation/hebcal_client.py) — unassigned to any dimension; it is the data source behind the digest/briefing chag + candle-lighting lines
The digest-briefing dimension found the erev-chag line is never emitted and the briefing has no Hebcal lines, but never examined hebcal_client itself: (a) is_chag/chagim_in_range filter category in {holiday, roshchodesh} and require yomtov flag, but candle-lighting for erev-chag comes from the 'candles' category which shabbat_times only reads on the weekly endpoint — so the data needed for the missing erev-chag line may not even be fetchable through the current API surface; (b) the 24h cache is keyed by ISO week/month and silently returns a stub {_stub:true} on fetch failure that callers may render as 'no chag' (a fail-quiet that suppresses a real holiday alert); (c) GEONAME_ID is 'nearest metro' so candle times are admittedly ~1min off — acceptable, but untested. The actual correctness of the chag/zman data feeding two surfaces was never probed.

### [medium] Timer concurrency on the live Google Sheet: finance(06:00) write, reminders-engine(07:25) write, digest(07:30) read+write, property(07:10/19:10) write all hit the same Sheet via gspread with no coordination
Four systemd timers mutate/read the one master Sheet within a 90-minute window with eventual-consistency semantics. The engine writes Last Sent/Status and recurrence bumps at 07:25; the digest reads the same tab at 07:30 and writes stamps; finance appends transactions at 06:00 whose SUMIFS the briefing/dashboard later read. gspread batch_update is not transactional and there is no lock or ordering guarantee beyond the timer clock (AccuracySec up to 1m on finance). A slow 07:25 run overlapping the 07:30 read, or a Sheets propagation lag, can make the digest read a half-written state. The outbox-budget dimension addressed the local ledger race; nothing addressed the multi-timer Sheet read-after-write race.

### [low] Dashboard finance reads depend on Sheet-computed columns (Actual / % of Target SUMIFS) being recomputed before the read
